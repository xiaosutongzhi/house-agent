# 偏好标注维度表（Rubric）

适用对象：人工标注 / LLM-judge / 自动化规则 judge

目标：为同一条用户问题（prompt）下的多个候选回答，稳定地产生 `chosen`（更好）与 `rejected`（更差），并与统一输出 schema 对齐。

## 0. 输出 Schema（硬约束）
回答必须是**严格 JSON**（不能有任何前后缀、解释文字、Markdown 代码块）。

必须包含且仅包含以下字段（顺序不限）：
- `greeting`: string
- `answer`: string
- `evidence`: array of string
- `next_question`: string

> 判定建议：
> - 只要 JSON 解析失败，直接判为 **不合格**（强烈倾向 rejected）。
> - 字段缺失、类型不对（例如 evidence 不是数组）也视为不合格。

## 1. 评分维度（建议总分 10 分）

### A. 格式与可解析性（0/2 分，权重最高）
- 2 分：可被 `json.loads` 解析；字段齐全；类型正确；没有多余文本。
- 1 分：可解析但存在轻微问题（例如多了无关字段、字段类型勉强可修复）。
- 0 分：不可解析 / 缺字段 / 类型明显错误。

### B. 字段质量（0~2 分）
- `greeting`
  - 要求：简短自然，不要太长的故事；能承接用户情绪/诉求。
- `next_question`
  - 要求：非空；与当前问题强相关；尽量是**单一明确问题**；用于补齐关键信息。

给分规则：
- 2 分：greeting 合理且 next_question 高相关、明确可执行。
- 1 分：有但一般（例如 next_question 太泛："方便说下预算吗" 但缺少针对性）。
- 0 分：缺失/空字符串/明显跑题。

### C. 证据与不胡编（0~2 分）
该项用于约束 `evidence`。

- 若 prompt **不提供 CONTEXT/引用材料**（例如只是用户提问，没有任何检索片段），则：
  - `evidence` 应为 `[]`（或非常谨慎地为空），且回答中不应出现“根据资料/根据 CONTEXT/官方规定写明”等指向不存在材料的说法。

- 若 prompt **提供了 CONTEXT**（例如包含 `CONTEXT:` 段落），则：
  - `evidence` 中每条都应能在 CONTEXT 中找到对应依据（最好是短引文或关键事实点）。
  - 不允许把 CONTEXT 没提到的细节当作证据（典型：编造学位、税费、年限、政策条款）。

给分规则：
- 2 分：evidence 使用符合场景（无 CONTEXT 则空；有 CONTEXT 则证据可对齐）。
- 1 分：证据基本克制但有轻微夸张/不可核验表达。
- 0 分：明显胡编 evidence、或在无 CONTEXT 情况下伪造“根据资料”。

### D. 回答有效性（0~2 分）
- 2 分：正面回应用户核心问题；结构清楚；给出可执行建议/解释；无明显事实性胡编。
- 1 分：大体相关但比较空泛（套话多、信息增量低）。
- 0 分：跑题/拒答/明显错误。

### E. 风格与风险控制（0~2 分，可选）
- 2 分：语气专业、不过度承诺；对不确定处有边界；不输出敏感个人信息。
- 1 分：基本正常但有夸张承诺或轻微不当表述。
- 0 分：包含明显不当承诺/引导违规/泄露隐私。

> 如果你只想“严格 JSON 格式优先”，可把 A、C、B 的权重提到更高（例如 A=4、C=3、B=2、D=1）。

## 2. `chosen` / `rejected` 选择规则
对同一 prompt 的候选回答集合：
1) 先按硬约束过滤：不可解析 / 缺字段 / 类型错 → 直接放入低优先级（倾向 rejected）。
2) 对可解析者按上述维度打分。
3) `chosen` = 得分最高者；`rejected` = 得分最低者。
4) 若分数接近或打平，优先级：
   - 证据不胡编（C） > 格式（A） > next_question 质量（B） > 回答有效性（D）。

## 3. 给 LLM-judge 的判分提示词模板（可直接用）

你可以把下面这段作为 judge prompt 的核心：

- 你将看到：用户问题（可能包含 CONTEXT），以及 2 个候选回答（均声称是 JSON）。
- 任务：根据 Rubric 逐项打分（A~D 至少），并给出：
  - 更好的回答是 A 还是 B（输出 `chosen=A` 或 `chosen=B`）
  - 简短理由（每项 1 句）
  - 若两者都不合格，仍需选相对更好的一个。
- 重要：如果无 CONTEXT，任何伪造 evidence 的回答必须强烈扣分。

## 4. 最小可自动化规则（用于 bootstrap）
如果你暂时不用 LLM-judge，也可以用如下自动规则产生偏好：
- JSON 可解析 + 字段齐全 + evidence 在无 CONTEXT 时为空 + next_question 非空 → 高分
- 非 JSON / 缺字段 / evidence 伪造 / next_question 为空 → 低分

这套规则能稳定把模型先推向“结构化可用”。
